# Patterns & Architectural Decisions

> Conventions to FOLLOW when editing, and the decisions behind them. Deviating silently is how bugs/drift get introduced.

## Identity & auth (suite-wide)

- **`X-User-Id` is a demo identity selector, NOT auth.** `get_current_user` (`server/app/middleware/auth.py:33`) resolves: JWT bearer ‚Üí API key ‚Üí `X-User-Id` header ‚Üí cookie ‚Üí `system` fallback. The header is client-supplied and trusted. Never expose OPA publicly without a real gate.
- **Demo gate** = shared-password HMAC token (`opa_demo_token` in localStorage), `Authorization: Bearer`. Code in `middleware/gate.py` is **present but NOT registered** in `main.py:125` (replaced by JWT path) ‚Äî verify which is live before relying on it.
- **RBAC is opt-in**: `require_app(app)` / `require_role(role)` deps. Frontends gate UI via `NoAccessGate` checking `user.apps.includes(app)` from `/api/users`. Both backend dep AND frontend gate must be satisfied ‚Üí if `apps[]` not seeded, user is blank-screened.
- **IAM is not an SSO provider** despite the name ‚Äî it's an admin UI over OPA's user table. Canonical-login (CROSS_APP_AUTH Phase 5) is unimplemented.
- Each app uses a DIFFERENT localStorage key for the actor (`iam.actorUserId`, `siu.currentUserId`, `claimguard.currentUserId`, `assistant_user_id`) ‚Äî a divergence source.

## Data / persistence

- **Single DB, `pipeline_mode` discriminator** (`post_pay`|`pre_pay`) hosts both PayGuard & ClaimGuard. FWA = disposition, not a mode. No multi-tenancy by design.
- **`server_default` rule**: any NOT NULL column a raw-SQL seed/X12 intake might omit MUST declare `server_default=` on the model (ORM `default=` alone is insufficient). Baseline migration is autogenerated from models, so declare on model + regenerate.
- **Alembic single squashed baseline**; `render_as_batch=True` + `naming_convention` in `database.py` (mandatory for SQLite ‚Äî unnamed constraints can't be reflected). Use `op.add_column` for plain adds; `batch_alter_table` only for drop/alter/constraint changes, with named constraints.
- **‚öÝÔ∏è Migrations currently DISABLED at startup** (`main.py:106-108`, "hanging") ‚Äî schema is NOT built in-process right now. CLAUDE.md claims the lifespan runs `alembic upgrade head`; the code disabled it. Verify before assuming fresh-DB build works.
- **Reference-data-first**: intake REJECTS unknown members/providers (`prepay_intake_service` ‚Üí `IntakeValidationError` ‚Üí 422). Don't auto-create reference rows at intake.
- Persistence intentionally ephemeral (SQLite, no volume); `SEED_ON_EMPTY=1` self-seeds empty deploys.

## Detectors & scoring

- **Orchestrator swallows per-detector exceptions** (`detectors/orchestrator.py`) ‚Äî a failing detector logs and continues. When findings are "missing," check logs for detector errors FIRST.
- `enabled_codes` + `score_multipliers` gate which detectors fire / rescale confidence (operator-tunable without code changes).
- **Confidence must be [0,1]** ‚Äî out-of-range silently distorts the Bayesian posterior.
- **Posterior** (`case_service._compute_posterior`): DET-08 fires ‚Üí 0.98 hard override (bypasses Bayes); no findings ‚Üí prior√ó0.50; else sequential `p ‚Üê p + (1-p)√óconfidence`. **DET-08 is special-cased twice** (exclusion check + posterior override) ‚Äî change both sites.
- **Priority** = `(amount_norm√ó0.60 + posterior√ó0.35 + urgency√ó0.05)√ó100`; bands ‚â•75 HIGH / 50-74 MED / <50 LOW. Weights come from `priority_config` DB (not function defaults).
- **Per-line attribution dedup** (`services/amount_at_risk.py` `compute_at_risk_deduped`) ‚Äî each claim line ‚Üí single highest-priority finding to avoid double-counting. New overlapping detectors must participate.
- `_DET_CODE_MAP` (`case_service.py`) aliases legacy IDs (`DUPLICATE_CLAIM_V1`‚Üí`DET-01`) ‚Äî always normalize through it.

## Assistant / MCP

- **Write-gate invariant**: the agent mutates state ONLY via `ask_user`/`present_view`/`confirm_action` UI tools (human-confirmed). All other MCP tools are read-only. Any new write tool MUST go through this gate.
- **ClearLink tools flow only to ClearLink** (see [[clearlink-tool-architecture]] memory) ‚Äî don't route ClearLink clinical tools through OPA's own data path; OPA proxies them.
- ClearLink tools are **DB-registered** (`agent_tools` rows) ‚Äî adding a tool = add a row (kind/sql_template/input_schema), not just code. (This is why `search_members` code is uncallable ‚Äî no row.)
- ClearLink agent tool calls audited to `agent_tool_calls`; OPA audit to `audit_logs` (case_id|claim_id).

## LLM config

- OPA: Anthropic SDK, models via `LLM_MODEL`/`ASSISTANT_MODEL` env; single config point `ai_service._client`/`MODEL`, reused by `document_generation_service`.
- ClearLink: TWO clients (SDK `claudeClient.js` default opus-4-8; axios calls default `ANTHROPIC_MODEL_SMART`=sonnet-4-6) ‚Äî inconsistent; pick one when touching.
- When working on anything LLM-shaped, consult the `claude-api` skill for current model IDs/pricing ‚Äî don't answer from memory.

## Frontend conventions

- React + Vite + TS + Tailwind; TanStack Query v5 for server state (siu/iam/assistant/claimguard); no global store.
- API base URL is **committed in `appUrls.ts`** (switched by `import.meta.env.PROD`), NOT env-driven ‚Äî except AppSwitcher which inconsistently reads `VITE_*_URL`. Changing backend URL = edit `appUrls.ts` + rebuild.
- Prod hosts: `*.penguinai.studio`, OPA = `payguard.penguinai.studio`.

## Audit & traceability (requirement f)

- OPA: immutable `audit_logs` per case/claim; pre-case events use `claim_id` only.
- ClearLink: `agent_tool_calls` (every MCP/tool call) + `audit_log` + `agent_runs`/`agent_actions`.
- **Decision log lives HERE** (`docs/codebase-map/`) + persistent memory (`MEMORY.md`). Record non-obvious design choices in memory with `type: project`/`feedback`.
