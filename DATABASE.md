## TL;DR — current state

- **DB:** SQLite at a **relative** path — `sqlite+aiosqlite:///./opa.db` (`app/config.py`). On Railway this lands in the container's **ephemeral** filesystem.
- **Schema is built by Alembic migrations** — the lifespan hook (`app/main.py` → `_run_migrations()`) runs `alembic upgrade head` on startup. One squashed baseline; `create_all` is gone.
- **No persistent volume** on Railway → **every deploy starts from an empty DB**, which migrations then build and `_seed_if_empty` populates. Reset-on-deploy = a clean demo (still intentional until Phase 2).
- **Auto-seed on deploy:** the lifespan hook seeds the demo dataset when `SEED_ON_EMPTY=1` **and** the DB is empty. `railway.toml` sets the flag; local dev and tests leave it off.
- **Every schema change needs a migration:** edit the model → `alembic revision --autogenerate -m "…"` → review → `alembic check` must be empty.

This is fine for a demo. It is **not** safe once real data must survive a deploy (needs Phase 2). The rest of this doc is the path to that.

## Two ways schema gets built — and why they collide

There are two independent mechanisms that can create schema, and running both against one DB is the root of every problem here.

| | `create_all_tables()` | Alembic migrations |
|---|---|---|
| What | `Base.metadata.create_all` — builds the **entire** current schema from the ORM models in one shot | A linked list of files in `server/migrations/versions/`, each evolving the schema one step (`upgrade()` / `downgrade()`) |
| Good for | Dev convenience, throwaway DBs, tests | **Production** — evolves an *existing* DB in place without dropping data |
| Can it ALTER an existing table? | **No** — only creates *missing* tables; never adds columns to existing ones | **Yes** — `ADD COLUMN`, backfill, etc. |
| Records what ran? | Nothing | The `alembic_version` table |

**The collision (historical — now resolved):** `create_all` built tables but left `alembic_version` stale, so a later `upgrade head` would try to replay `CREATE TABLE` for tables that already existed → `table already exists` → abort. This is why the dev DB was once stuck at `g7b8c9d0e1f2` while the schema was much newer. **Resolved by the cutover** (see status banner): `create_all` is gone, the chain is a single squashed baseline, and the DB is stamped at that baseline — so only one mechanism (migrations) ever touches the DB.

**Rule going forward:** migrations are the single source of truth. Never reintroduce a `create_all` call against a real DB.

## The `server_default` rule (important — it silently breaks seeding)

`create_all` only emits a SQL-level `DEFAULT` for a column declared with **`server_default=`**. An ORM-side **`default=`** is applied by SQLAlchemy *during ORM inserts only* — it produces **no** DB default.

The raw-SQL seeds (`seed/*.py`) and X12 intake insert rows with plain `INSERT` and sometimes **omit** columns, relying on a DB default. If such a column is `NOT NULL` and has only `default=` (no `server_default=`), a fresh `create_all` DB rejects the insert:

```
sqlite3.IntegrityError: NOT NULL constraint failed: <table>.<column>
```

This is what was silently breaking seed-on-deploy. Columns already fixed with `server_default`: `claims.pipeline_mode`, `opa_cases.review_time_minutes`, `opa_cases.law_enforcement_hold`, `opa_cases.siu_frozen` (plus the seed insert for `ml_training_config` was made explicit).

**When you add a `NOT NULL` column:** if any raw INSERT might omit it, give it `server_default=` in the model — not just `default=`. Migrations should declare the same `server_default`.

## Auto-seed on startup (`_seed_if_empty`)

- Lives in `app/main.py`. Runs the full seed **only if** `settings.seed_on_empty` is true **and** `opa_users` is empty.
- **Idempotent:** a warm restart that still has data skips seeding (logs `DB already populated — skipping seed`).
- **Off by default** (`SEED_ON_EMPTY` unset) so local dev and the test suite (both fire the lifespan via `TestClient`) never auto-seed.
- `railway.toml` sets `SEED_ON_EMPTY=1` in the start command, so each ephemeral deploy self-seeds.
- Runs the synchronous, CPU-heavy seed in a worker thread; failures are logged but never crash startup.
- Points the seed's `DB_PATH` at the same SQLite file the app reads, so they can't diverge.

## Validating persistence on Railway

SQLite is **not** a managed service — it's a file in your app container, and that filesystem is ephemeral by default. Persistence requires a **Volume** *and* the DB file written onto it. To check:

1. **Volume attached?** Dashboard → service → **Settings → Volumes**, or `railway volume list`. When mounted, Railway injects `RAILWAY_VOLUME_MOUNT_PATH` / `RAILWAY_VOLUME_NAME` (`railway variables`).
2. **DB file on the volume?** Our path is relative (`./opa.db` → ephemeral layer). A volume only persists files **under its mount path**, so `database_url` must point into it, e.g. `sqlite+aiosqlite:////data/opa.db`. Check both conditions.
3. **Managed Postgres instead?** `railway variables` shows a `postgres://…` `DATABASE_URL`. Note the app only uses `database_url` from `config.py` — a provisioned Postgres does nothing until the app is configured to read it.
4. **Definitive empirical test:** create a record → trigger a redeploy or restart → look for it. **Gone = ephemeral. Survived = persistent.**

Current expectation: **ephemeral** (no volume config in `railway.toml`, relative SQLite path).

## Production path — step by step

Do these **in order**. Each phase is a prerequisite for the next.

### Phase 0 — Decide it's time
Trigger: real data must survive a deploy (first real users, persisted audit trail, etc.). Until then, demo mode is the right, cheap choice — don't pre-build this.

### Phase 1 — Squash migrations into a validated baseline
The current chain is broken, so reconcile it once by making the **current models** the baseline.

```bash
cd server
rm migrations/versions/*.py              # delete the stale chain
# Point alembic at an EMPTY DB so the diff is "create everything":
alembic revision --autogenerate -m "baseline schema"
#   → edit the new file: set  down_revision = None   (it's the new root)
```

**Validate** (SQLite autogenerate has blind spots — CHECK constraints, some
`server_default`s, batch-mode quirks):

```bash
# Replay baseline into a throwaway DB:
rm -f /tmp/baseline.db
DATABASE_URL=sqlite+aiosqlite:////tmp/baseline.db alembic upgrade head
# Compare its schema to what create_all (the models) produces, and diff.
# They MUST match. Hand-patch the baseline until they do.
```

Because the baseline is generated *from* the models, it matches dev by construction — the historical model-vs-migration drift disappears at this point.

### Phase 2 — Make the DB persistent
Pick one:

- **A. Railway Volume + SQLite.** Attach a volume (e.g. mount `/data`), set `database_url` (or `DATABASE_URL`) to `sqlite+aiosqlite:////data/opa.db`. Cheapest; single-writer only.
- **B. Railway Postgres (conventional).** Add the Postgres plugin; read its injected `DATABASE_URL`; swap the driver `aiosqlite` → `asyncpg` (a config-level change). Better for concurrency/backups.

Re-run the **empirical persistence test** above to confirm data now survives a deploy.

### Phase 3 — Make migrations the source of truth in deploy
- Add a release/start step **before** uvicorn: `cd server && alembic upgrade head`.
- **Gate `create_all` out of production** (e.g. only call `create_all_tables()` when `environment != "production"`), so it can't race migrations and recreate the collision.
- Decide how prod gets seed/reference data — a one-time job, **not** on every boot. Reconsider `SEED_ON_EMPTY` once the DB persists (you don't want to reseed a populated prod DB).

### Phase 4 — Retire `create_all` everywhere + add a CI drift guard
- There is exactly **one** `create_all_tables()` call site (the lifespan hook) and tests don't depend on it, so removal is a single-point change — but only **after** Phases 1–3, since `create_all` is currently the only *working* schema builder.
- Removing `create_all` loses its free "schema always matches the models" guarantee. Replace it with a **CI guard**: run `alembic upgrade head` then `alembic check` (or an autogenerate that must produce an **empty** diff). If a model changed without a migration, CI fails. This is what keeps "migrations are the source of truth" honest.

### End state
- Dev, CI, and prod all build schema the same way: `alembic upgrade head`.
- Every schema change → `alembic revision --autogenerate` + review.
- CI guard prevents model/migration drift.
- `create_all` is gone; persistence is real.
